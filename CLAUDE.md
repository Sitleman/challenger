# CLAUDE.md — процесс разработки challenger

Язык общения: **русский**.

Полный план проекта — в `README.md` (фазы: Спайк №0 → Фундамент → MVP → v2 → v3).
Этот файл описывает **как** мы работаем, а не что делаем.

---

## Роли

- **Пользователь** — заводит issue с коротким описанием и своим видением. Держит два гейта: апрув плана и merge.
- **Claude Code** — прорабатывает issue, пишет код, гоняет саморевью, открывает PR.

## Гранулярность

**Issue = 1 PR = один мержибельный кусок ценности.** Не дробим до отдельных функций, не раздуваем до целой фазы.

## Deploy

Руками (один Go-бинарь + Postgres). CI/CD — отдельный issue после MVP.

## 🔄 Алгоритм на каждый issue

```
1. ВХОД: пользователь даёт issue (короткое описание + видение)

2. ПРОРАБОТКА (моя зона)
   ├─ прогоняю skill brainstorming по конкретному куску
   └─ выдаю ОДНО из двух:
      • ПЛАН — техпроработка: таблицы/эндпоинты/файлы, edge-cases,
        Definition of Done. Кладу в комментарий к issue.
      • ВОПРОСЫ — если правильное решение неочевидно, спрашиваю
        ДО написания плана (AskUserQuestion на реальных развилках).

3. АПРУВ плана          ── гейт №1: пользователь ──
   └─ не пишу код, пока план не апрувнут.

4. РЕАЛИЗАЦИЯ (моя зона)
   ├─ ветка CH-<issue>-<short-description-en>
   ├─ TDD: тест → код → рефактор (skill test-driven-development)
   └─ коммиты в формате CH-<issue>: <short description en>

5. САМОРЕВЬЮ (моя зона)
   ├─ /code-review на диффе → правлю найденное
   ├─ линтеры чистые (golangci-lint / eslint+prettier)
   └─ verification-before-completion: тесты зелёные, проверено руками

6. PR (моя зона)
   └─ открываю PR: title `CH-<issue>: <short description en>`,
      описание — что сделано в 1–5 предложениях на русском, `Closes #N`

7. РЕВЬЮ + MERGE      ── гейт №2: мержит только пользователь ──
   └─ squash-merge, ветку удаляем

8. DEPLOY — руками (позже CI)
```

**Два человеческих гейта:** апрув плана (шаг 3) и merge (шаг 7). Между ними работаю автономно.

---

## Правила проработки issue (шаг 2)

- Если решение однозначно → сразу **план**, без лишних вопросов.
- Если есть развилка, меняющая архитектуру/UX/данные → сначала **вопросы**
  (`AskUserQuestion`), потом план. Не гадаю на важных решениях.
- План всегда содержит **Definition of Done** — по нему проверяем готовность.
- Изменения схемы БД (`Challenge / Participant / Day / Idea`) явно отмечаю в плане.

## 📐 Конвенции

### Язык
- **Документация** (README, docs) — русский.
- **Общение** — русский.
- **Код** (идентификаторы, строки, коммиты, ветки, PR-title) — английский.
- **Комментарии в коде не пишем.** Только в очень редких случаях, когда без этого никак — тогда на английском.

### Стиль кода
- **Функции и методы сортируем в порядке вызова** (stepdown-правило): вызывающая — выше вызываемой. Файл читается сверху вниз по потоку вызовов: сначала высокоуровневая функция, затем те, что она вызывает.

### Git
- **Коммит:** `CH-<issue>: <Short description on english 1-10 words>`
- **Ветка:** `CH-<issue>-<short description on english 1-5 words>`
- **PR title:** `CH-<issue>: <Short description on english 1-10 words>`
- **PR description:** что сделано в этом PR, 1–5 предложений на русском. В конце `Closes #N`.
- **Merge:** squash (одна фича = один коммит в master), ветку после merge удаляем.

### Definition of Done (для каждого PR)
- Тесты зелёные.
- Линтер чистый.
- Миграция обратима (up/down).
- Проверено руками.
- Отмечен соответствующий чекбокс в README.

### Решения (ADR)
Важные развилки (результат Спайка №0 план А/Б и т.п.) фиксируем короткими
записями в `/docs/decisions/`: контекст → решение → почему.

### API-контракт
- Префикс `/api/`, middleware валидации `initData` на защищённых роутах.
- Формат ответа: `{data}` / ошибки `{error: {code, message}}`.
- Источник истины типов — Go-структуры; фронт держит зеркальные TS-типы.

## 🗂 Структура репо (монорепо)

```
/cmd/challenger/       — main.go (бот + API в одном бинаре)
/internal/bot/         — telego-хендлеры
/internal/api/         — chi-роуты
/internal/db/          — sqlc: queries + сгенерированный код
/migrations/           — SQL-миграции (goose, NNNN_description.sql)
/web/                  — React + Vite + TS
/docs/decisions/       — ADR
docker-compose.yml     — Postgres для локалки
Makefile               — make run / test / migrate / sqlc
.env.example           — ключи без значений (.env в .gitignore)
```

## Спайк №0 — вне процесса

Разведка `message_reaction` — эксперимент на полдня без PR-ceremony.
Делаем первым, результат (план А: реакции / план Б: кнопка-аппрув) фиксируем
в `/docs/decisions/` — он влияет на все issue про подтверждение дня.

## Инфраструктура (первый инфра-issue)

- **CI на тесты** (не деплой): GitHub Actions гоняет `go test` + линтеры на каждый PR — pre-merge защита.
- Миграции: **goose**. Применённую миграцию не редактируем — только новая сверху.
- Секреты: только через env, `.env` в `.gitignore`, `.env.example` в репо.
