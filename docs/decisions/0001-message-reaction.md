# 0001 — Подтверждение дня: реакции (`message_reaction`)

**Дата:** 2026-07-19
**Статус:** принято
**Спайк:** #1

## Контекст

Механика зачёта дня зависит от того, доходит ли до бота персональный апдейт
`message_reaction` при реакции на сообщение в группе — или Telegram отдаёт только
анонимный агрегат `message_reaction_count` (без идентичности реагирующего и без
дельты). Во втором случае вести счётчик уникальных реакций невозможно и пришлось
бы делать план Б (кнопка-аппрув внутри аппа).

Требования Telegram: бот — **админ** группы, `allowed_updates` явно содержит
`message_reaction` (в дефолтный набор он не входит).

## Проверка

Одноразовый бот (`spike/message-reaction/`) на telego, long polling с
`AllowedUpdates: ["message", "message_reaction"]`, в тестовой супергруппе
«Monthly challenges» (`is_forum: true`, бот — админ). Реакции ставились/менялись/
снимались, каждый апдейт логировался целиком.

Результат — приходил именно `message_reaction` (персональный), `message_reaction_count`
не пришёл ни разу:

| Действие | `old_reaction` | `new_reaction` | Трактовка |
|---|---|---|---|
| Поставил 🔥 | `[]` | `[🔥]` | +1 |
| Поставил ❤ | `[]` | `[❤]` | +1 |
| Сменил ❤ → 💯 | `[❤]` | `[💯]` | смена (счётчик не меняется) |
| Снял 💯 | `[💯]` | `[]` | −1 |

Пример апдейта:

```json
{
  "chat": { "id": -1003436893374, "type": "supergroup", "is_forum": true },
  "message_id": 244,
  "user": { "id": 532029049, "username": "grv_nikita" },
  "old_reaction": [],
  "new_reaction": [ { "type": "emoji", "emoji": "🔥" } ]
}
```

Апдейт содержит всё необходимое: `message_id` (связь с днём/пруфом),
`user.id` (уникальность реагирующего), `old/new_reaction` (дельта +1/−1/смена).

## Решение

**План А — зачёт по реакциям.** Счётчик реакций ведём сами в БД, пересчитывая
по дельте `old_reaction`/`new_reaction` на событие `message_reaction`.

## Следствия

- В боевом боте обязательно: бот — админ группы, `AllowedUpdates` включает
  `message_reaction`.
- Счётчик считаем по **уникальным** `user.id` на `message_id` (смена эмодзи тем же
  юзером счётчик не увеличивает).
- Не проверено (принято как допущение, риск низкий): в очень больших группах
  Telegram может присылать анонимный `message_reaction_count` вместо персонального —
  для нашей небольшой команды неактуально.
- Одноразовый код спайка (`spike/message-reaction/`) можно удалить после Фундамента.
- Технический нюанс окружения: `api.telegram.org` резолвился только по IPv6 →
  fasthttp-клиенту telego нужен dual-stack дайлер (`fasthttp.DialDualStack`).
